Feature 1 — Tool-Execution Gate with Cryptographic Approval Token

"A tool-execution gate requiring an approval token cryptographically bound to an action-plan hash before an external-write operation is executed."

This narrative describes a concrete application of Feature 1 as implemented within Higgins.ai (higginsworkassistant.com), demonstrating novelty and non-obviousness over the eight closest prior-art references identified in the Feature 1 screening analysis.

1. Background — The Security Gap

Higgins operates across 60+ regions, servicing 42,232+ associated businesses (Slide 09), integrating with SAP, Teamcenter, Veeva, and Trackwise (Slide 08). When an AI agent executes an unverified external-write operation — even one that has drifted slightly from its approved form — this poses significant compliance, data integrity, and legal risk. Feature 1 addresses this with a novel cryptographic binding mechanism that no prior art reference teaches.

2. Implementation — The Five-Stage Gate Pipeline

Stage 1
Canonical Record
Request ID + Tenant ID + vector embeddings generated on intake
Stage 2
Write-Semantics Tag
Each tool call classified READ_ONLY / INTERNAL_WRITE / EXTERNAL_WRITE
Stage 3
Hash at Review
Deterministic SHA-256 hash of complete action plan at moment of human review
Stage 4
Approval Token
Token = signature over hash + timestamp + tenant ID + user ID
Stage 5
Gate Validation
4-condition check immediately before every EXTERNAL_WRITE fires
Concrete Example — Higgins Biomedical: A QA associate requests a deviation investigation update to Trackwise. The action plan contains a READ_ONLY retrieval and an EXTERNAL_WRITE commit. The plan is hashed, presented for review, and an approval token is issued. At execution, the gate validates: (i) token present and valid; (ii) hash recomputed; (iii) hashes match; (iv) tenant context matches. If any condition fails, execution is blocked and an alert is logged.

Feature 2 — Hybrid Request-Classification Engine

"A hybrid request-classification engine combining a rules-based classifier and a statistical classifier, with merged outputs using weighted scoring, voting, or hierarchical precedence, to assign domain/task labels for routing."

Feature 2 is the gateway through which every user request enters the Higgins orchestration layer. Its accuracy directly determines whether the platform delivers on its core promise: that 101+ AI models working in specialized teams outperform any single generalist AI (Slide 04).

1. Background — The Classification Problem

Higgins serves 42,232+ associated businesses across 60+ regions (Slide 09) spanning biomedical, pharmaceutical, real estate, financial services, insurance, and legal. A single-classifier routing system fails predictably: rules-based systems cannot handle semantic ambiguity; statistical systems cannot enforce hard compliance constraints in regulated industries. Feature 2 solves this with a hybrid architecture that no prior art reference teaches.

2. Implementation — The Three-Layer Hybrid Pipeline

Layer 1 — Rules-Based
Domain Flag Detection
Deterministic pattern-matching on FDA nomenclature, ISO/ASTM identifiers, GxP terminology, MLS vocabulary. Fast, compliance-safe, binary domain signal + flag-confidence score.
Layer 2 — Statistical
Embedding Confidence
Cosine similarity between request embedding and pre-trained domain centroid embeddings. Handles semantic variation, abbreviations, cross-domain requests, novel phrasing.
Layer 3 — Merge
Weighted Score Fusion
W₁ × rules score + W₂ × embedding score. W₁ elevated in regulated domains (Biomedical); W₂ elevated in general-purpose domains. Final domain + task label assigned.
Worked Example — Higgins Biomedical: Request: "Help me write up the investigation for the failed pH test on Batch 2024-0317." Rules classifier fires on "investigation," "failed," "pH test," "Batch" → flag score: 0.91 (Biomedical). Statistical classifier: Biomedical 0.84, General Document 0.31, Realty 0.02. Weighted fusion (W₁=0.6, W₂=0.4): final score = 0.6×0.91 + 0.4×0.84 = 0.882 → BIOMEDICAL / DEVIATION-INVESTIGATION. Routed to Higgins Biomedical team with FDA + ISO/ASTM access.

"Generation of a JSON-based action plan for tool execution, including write-semantics fields classifying operations as read-only, internal-write, or external-write, and including deadlines, concurrency limits, and resource-allocation constraints."

Feature 4 — Citation-Enforced RAG with Tenant-Isolated Vector Stores & Reject/Regenerate Loop RANKED #2 STRONGEST

"Retrieval-augmented inference that retrieves evidence from tenant-isolated vector stores and external sources, enforces citation pointers in generated responses, and rejects/regenerates candidate responses lacking required citations."

Feature 4 is the technical embodiment of Dr. Hakim's principle from the "Transformation" book: "If your AI hallucinates, provides shallow answers, gives you too much when you need less — then AI becomes a burden, not a gift." Feature 4 makes hallucination architecturally impossible to deliver to the user.

1. The Five-Stage Citation-Enforced RAG Pipeline

Stage 1
Evidence Retrieval
Query tenant-isolated vector store (primary) + domain-validated external sources (secondary)
Stage 2
Evidence Packet
Canonical JSON packet: chunk_id + source_id + timestamp + relevance_score per chunk. Immutable once assembled.
Stage 3
Constrained Generation
Lead AI generates candidate response with mandatory inline citation pointer per factual claim
Stage 4 — GATE
Citation Validation
Platform validates every citation pointer against canonical evidence packet. Reject if missing or invalid.
Stage 5
Deliver or Regenerate
Pass → deliver. Fail → incremental retrieval + regeneration loop until all citations present.

Feature 5 — Immutable Audit Ledger: Request Records, Action-Plan Hashes, Approval Tokens & Execution Receipts

#5
Ranked #5 of 5 in overall feature strength — but the screening analysis explicitly notes it is "strongest when tied to pre-execution authorization rather than generic audit logging." Feature 5 is drafted as a dependent claim on Feature 1's cryptographic gate, creating a compound limitation that no prior art reference anticipates.
"An audit ledger storing request records, action-plan hashes, approval tokens, and execution receipts to support traceability, replay, and forensic review."

1. The Four-Field Audit Tuple

Field 1
Request Record
Canonical request record: request ID, tenant ID, domain label, task label, user ID. Links audit event to original user intent.
Field 2
Action-Plan Hash
SHA-256 hash of JSON action plan at moment of human review. Proves plan was not modified between approval and execution.
Field 3
Approval Token
Token ID + approved_by + approval_timestamp + cryptographic_signature + hash_at_approval. Full authorization record.
Field 4
Execution Receipt
Tool name + write_semantics + gate_outcome (PERMITTED/BLOCKED) + execution_timestamp + external_confirmation_id + hash_verified.
